git clone

リモートリポジトリをローカルにコピーする
git clone は GitHub などのリモートリポジトリをあなたのPCにまるごとコピーするコマンドです。履歴も含めて全データが手元に来ます。
📋 CONCEPT
リモートリポジトリ（GitHub）→ ローカルリポジトリ（自分のPC）へのコピーを clone と呼びます。このワークショップでは、チームメンバー全員が同じリポジトリをクローンして進めます。
  • ✓
    Sourcetree を起動する
    アプリケーションから Sourcetree を開いてください。
  • ✓
    「新規」→「URLからクローン」を選択する
    左上の ＋新規リポジトリ または File > New から「URLからクローン」を選択します。
  • ✓
    リポジトリのURLを入力する
    「ソースURL」欄に講師から共有された GitHub リポジトリの URL を貼り付けます。
  • ✓
    保存先を確認して「クローン」をクリック
    保存先パスを確認し（例：~/Documents/workshop）、クローン ボタンを押します。
  • ✓
    ファイルが手元にコピーされたことを確認する
    Sourcetree にリポジトリが表示され、top-page.html が存在することをファインダー／エクスプローラーで確認します。
💡 TIP
クローン完了後、Sourcetree の左サイドバーに main ブランチが表示されます。これがデフォルトブランチです。
🌿

git checkout

新しいブランチを作成して切り替える
git checkout は作業ブランチを切り替えるコマンドです。新機能や修正は必ず新しいブランチを作って作業します。これがチーム開発の鉄則です。
📋 CONCEPT
ブランチ はコードの「作業コピー」。main から分岐して作業することで、他の人の作業を邪魔せずに開発できます。このワークショップでは 2人が同じファイルの同じ箇所を別ブランチで編集 してコンフリクトを意図的に起こします。
  • ✓
    現在 main ブランチにいることを確認する
    左サイドバーの main の横に ● が表示されていることを確認します。
  • ✓
    自分専用のブランチを作成する
    上部ツールバーの ブランチ をクリック。ブランチ名を 自分の名前-conflict-resolve と入力して「ブランチを作成」します。
    例：yamada-conflict-resolve
  • ✓
    新ブランチに切り替わったか確認する
    左サイドバーに 自分の名前-conflict-resolve が太字で表示されていることを確認します。
  • ✓
    ペアの相手も同じ手順でブランチを作ったか確認する
    2人1組の場合、お互いに異なるブランチ名（例：yamada-conflict-resolve と sato-conflict-resolve）が作れていることを確認します。
⚠️ POINT
このワークショップでは「2人が同じ top-page.html の同じ行を別々に編集する」ことでコンフリクトを再現します。次のステップでどこを編集するか、ペアと相談しながら進めましょう。
📝

git commit

top-page.html を編集してコミットする
ここがコンフリクトの仕込みポイントです。 2人が 同じファイルの同じ行 を異なる内容で編集することで、後のステップでコンフリクトが発生します。
🎯 今回の作戦
まずはリポジトリ内の top-page.html を元に、自分用のコピー top-page.your-name.html を作成して使います。その上で 2人とも top-page.your-name.html の キャッチコピー（編集ポイント C） を、それぞれ別の文言に書き換えます。同じ行が2つの異なる内容になるため、マージ時にコンフリクトが起きます。
  • ✓
    top-page.html をコピーして top-page.your-name.html を作成する
    リポジトリ内の top-page.html を VS Code などのエディタで開き、 名前を付けて保存 などを使って top-page.your-name.html という名前で同じフォルダにコピーを作成します。 以降の編集はこのファイルで行います。
  • ✓
    top-page.your-name.html をエディタとブラウザで開く
    top-page.your-name.html を VS Code で開きます。同時にブラウザにドラッグして現在の見た目を確認しましょう。
  • ✓
    【編集ポイント C】キャッチコピーを自分の文言に変更する
    <!-- 編集ポイント C --> 直下の <h1> 内のキャッチコピーを書き換えます。
    ペア A 例：未来を、チームの力で切り拓く。
    ペア B 例：挑戦を、仲間と共に歩む。
    ※ 2人が必ず異なる文言にすることがポイントです！
  • ✓
    【編集ポイント F】自分の名前・役職を追加する
    メンバーカードの 山田 太郎 を自分の名前に、Team Lead を自分の役職に書き換えて保存します。
  • ✓
    Sourcetree で差分を確認する
    作業ツリーのファイル に top-page.your-name.html が表示されます。クリックすると右側に差分（赤＝削除、緑＝追加）が確認できます。
  • ✓
    ステージングしてコミットする
    ＋ ステージング → コミットメッセージ入力 → コミット ボタンを押します。
    例：Update catchcopy and add member info
💡 GOOD COMMIT MESSAGE
✅ Update catchcopy and add member info
❌ 修正した / html変えた
🚀

git push

自分のブランチをリモートに送信する
git push で、ローカルのコミットを GitHub に送ります。2人ともプッシュすることで、次のステップでコンフリクトが発生する準備が整います。
📋 CONCEPT
コミットはローカルだけに存在します。push することで初めてリモートに反映され、他の人と共有できます。
  • ✓
    ヒストリーでコミットを確認する
    Sourcetree の ヒストリー タブを開き、先ほどのコミットが表示されていることを確認します。
  • ✓
    「プッシュ」ボタンをクリックする
    上部ツールバーの プッシュ をクリックします。
  • ✓
    自分のブランチを確認してプッシュする
    ダイアログで 自分の名前-conflict-resolve が選択されていることを確認して「プッシュ」をクリックします。
  • ✓
    GitHub でブランチが追加されたか確認する
    ブラウザでリポジトリを開き、自分の名前-conflict-resolve ブランチが表示されることを確認します。
  • ✓
    ペアの相手もプッシュを完了したか確認する
    GitHub 上に2人分のブランチ（yamada-conflict-resolve と sato-conflict-resolve など）が揃っていることを確認します。
GitHub で2人分のブランチが並んでいるのを確認してスクリーンショットを撮っておくと振り返りに使えます。
⬇️

git pull / merge

相手のブランチを main にマージしてコンフリクトを発生させる
いよいよコンフリクトを意図的に起こします。ペアのどちらか 1人 がこのステップを担当して、2つのブランチを main にマージしましょう。
🎯 このステップのシナリオ
担当者A が自分のブランチを main にマージ → 続けて 担当者A が担当者Bのブランチも main にマージしようとする → 同じ行が違う内容 なのでコンフリクトが発生！
  • ✓
    main ブランチに切り替える
    左サイドバーの main をダブルクリックして切り替えます。
  • ✓
    プルして main を最新の状態にする
    上部ツールバーの プル をクリックし、origin/main から最新を取得します。
  • ✓
    自分のブランチを main にマージする（1人目）
    上部ツールバーの マージ をクリック → 自分の名前-conflict-resolve を選択して「OK」を押します。
    この時点ではコンフリクトは発生しません。
  • ✓
    相手のブランチを main にマージする（2人目を取り込む）
    もう一度 マージ をクリック → 相手の名前-conflict-resolve を選択して「OK」を押します。
    ここで コンフリクトが発生 します！
  • ✓
    Sourcetree でコンフリクトを確認する
    作業ツリーのファイル に top-page.html が ！（エクスクラメーション）マーク つきで表示されます。これがコンフリクトのサインです。
Sourcetree に「！」マークがついた top-page.html が表示されたらコンフリクト発生成功です。次のステップで解消します。
⚡

コンフリクト解消

競合した変更を手動で解決してコミットする
コンフリクトはエラーではありません。「どちらの変更を残すか」を人間が判断する 作業です。焦らず、ファイルの中身を確認しながら進めましょう。
📋 コンフリクトマーカーとは？
Gitはコンフリクトが起きた箇所を以下のような記号で囲んで教えてくれます。このマーカーを手動で削除して、残したい内容だけにするのがコンフリクト解消です。
top-page.html — コンフリクトが発生した箇所のイメージ
  <div class="hero">
<<<<<<< HEAD（現在のブランチ＝自分の変更）
    <h1>未来を、チームの力で切り拓く。</h1>
======= （ここが境界線）
    <h1>挑戦を、仲間と共に歩む。</h1>
>>>>>>> sato-conflict-resolve（相手のブランチ＝相手の変更）
  </div>

解消方法は3通りあります：

🔵

自分の変更を採用

HEAD（青い行）だけを残し、マーカーと相手の行を全部削除する

🟢

相手の変更を採用

相手（緑の行）だけを残し、マーカーと自分の行を全部削除する

🟡

両方を合体させる

両方の行を残して意味が通るよう手動で編集する

  • ✓
    VS Code で top-page.html を開く
    Sourcetree の ！ がついた top-page.html を右クリック →「外部エディタで開く」または直接 VS Code で開きます。
  • ✓
    コンフリクトマーカーを見つける
    <<<<<<< HEAD を検索（Ctrl/Cmd+F）して、コンフリクトが起きている行を特定します。VS Code ではコンフリクト箇所に 青と緑のハイライト と「Accept Current Change」などのボタンが表示されます。
  • ✓
    残す内容を決めてマーカーを削除する
    VS Code のボタン（Accept Current Change / Accept Incoming Change / Accept Both Changes）をクリックするか、手動で <<<<<<<・=======・>>>>>>> の行を削除して保存します。
  • ✓
    ブラウザで top-page.html をリロードして確認する
    コンフリクトマーカーが残っていると HTML が壊れます。ブラウザで正常に表示されることを確認しましょう。
  • ✓
    Sourcetree でコンフリクト解消済みにする
    Sourcetree の ！ がついたファイルを右クリック →「競合を解決済みとしてマーク」を選択します。
  • ✓
    ステージングしてコミットする
    top-page.html を ＋ ステージング → コミットメッセージを入力 → コミット を押します。
    例：Resolve conflict in top-page.html catchcopy
  • ✓
    main ブランチに push する
    上部ツールバーの プッシュ → main ブランチを選択して「プッシュ」をクリックします。
💡 コンフリクト解消のコツ
マーカー（<<< === >>>）は必ずすべて削除しましょう。1行でも残っていると HTML が壊れます。保存後は必ずブラウザで動作確認してください。
⚠️ よくあるミス
コンフリクトを解消せずに git add してしまうとマーカーが混入したままコミットされます。必ずブラウザで確認してから ステージングしましょう。
