git clone

リモートリポジトリをローカルにコピーする
git clone は、GitHub などのリモートリポジトリをあなたのPC上にまるごとコピーするコマンドです。 これがプロジェクトへの参加の第一歩。Sourcetree では数クリックで完了します。
📋 CONCEPT
リモートリポジトリ（GitHub上）→ ローカルリポジトリ（あなたのPC）にコピーすることを clone（クローン） と呼びます。履歴も含めて全てのデータが手元に来ます。
  • ✓
    Sourcetree を起動する
    アプリケーションから Sourcetree を開いてください。
  • ✓
    「新規」→「URLからクローン」を選択する
    左上の ＋新規リポジトリ または File > New から「URLからクローン」を選択します。
  • ✓
    リポジトリのURLを入力する
    「ソースURL」欄に講師から共有された GitHub リポジトリのURLを貼り付けます。
  • ✓
    保存先フォルダを確認して「クローン」ボタンをクリック
    保存先パスを確認し（例：~/Documents/workshop）、クローン ボタンを押します。
  • ✓
    ファイルが手元にコピーされたことを確認する
    Sourcetree にリポジトリが表示され、ファインダー／エクスプローラーでファイルが存在することを確認します。
💡 TIP
クローンが完了すると、Sourcetree の左サイドバーに main（または master）ブランチが表示されます。これがデフォルトブランチです。
🌿

git checkout

新しいブランチを作成して切り替える
git checkout は作業するブランチを切り替えるコマンドです。 本番コードを守るため、新機能や修正は必ず新しいブランチを作って作業します。これがチーム開発の鉄則です。
📋 CONCEPT
ブランチ とは、コードの「作業コピー」のようなもの。main ブランチから分岐して作業することで、他の人の作業を邪魔せずに開発できます。
  • ✓
    左サイドバーの「ブランチ」を確認する
    現在 main ブランチにいることを確認しましょう。ブランチ名の横に ● が表示されます。
  • ✓
    新しいブランチを作成する
    上部ツールバーの ブランチ ボタンをクリック。ブランチ名を feature/yourname と入力して「ブランチを作成」します。
  • ✓
    新しいブランチに切り替わったか確認する
    左サイドバーに feature/yourname が表示され、太字（現在のブランチ）になっていることを確認します。
  • ✓
    ダブルクリックで別のブランチに切り替えてみる
    左サイドバーの main をダブルクリック → また feature/yourname に戻ります。これが checkout の操作です。
💡 TIP
ブランチ名の命名規則は会社やプロジェクトによって異なりますが、feature/機能名 や fix/バグ名 のように種類を prefix につけるのが一般的です。
⚠️ CAUTION
作業を始める前に必ず正しいブランチにいることを確認しましょう。間違ったブランチで変更してしまうと取り返しが大変です。
📝

git commit

変更をローカルリポジトリに記録する
git commit は、ファイルへの変更を「スナップショット」として保存するコマンドです。 コミットは変更の履歴として残り、いつでも過去の状態に戻れるようになります。
📋 CONCEPT
コミットは ステージング（add） → コミット（commit） の2段階で行います。ステージングは「この変更をコミットに含める」という宣言です。
📂 使用ファイル
リポジトリ内の top-page.html を使います。会社・チームの紹介ページです。ファイル内の <!-- 編集ポイント A〜F --> コメントを目印に変更箇所を見つけましょう。
  • ✓
    top-page.html をエディタとブラウザで開く
    top-page.html を VS Code などのエディタで開きます。同時にブラウザにドラッグ＆ドロップして、変更前の見た目を確認しておきましょう。
  • ✓
    【編集ポイント B】チーム名・会社名を変更する
    <!-- 編集ポイント B --> の直下にある Team Horizon を自分たちのチーム名に変更して保存します。
    例：Dev Squad Tokyo
  • ✓
    【編集ポイント F】自分の名前・役職を追加する
    <!-- 編集ポイント F --> の下にあるメンバーカードの 山田 太郎 や役職 Team Lead を自分の名前・役職に書き換えて保存します。
  • ✓
    【編集ポイント A】メインカラーを変更する
    ファイル上部の CSS 変数 --color-primary: #2563eb; の色コードを変えて保存します。
    例：緑 → #16a34a　紫 → #7c3aed　赤 → #dc2626
    ブラウザをリロードしてナビやボタンの色が変わったか確認しましょう。
  • ✓
    Sourcetree で差分を確認する
    Sourcetree の 作業ツリーのファイル に top-page.html が表示されます。ファイルをクリックすると右側に差分（赤＝削除した行、緑＝追加した行）が表示されます。
  • ✓
    ステージングしてコミットする
    top-page.html を ＋ ステージング してから、変更内容がわかるコミットメッセージを書いて コミット ボタンを押します。
    例：Update team name, member info, and primary color
💡 GOOD COMMIT MESSAGE
良いコミットメッセージは 「何を」変えたか が1行でわかるもの。
✅ Update team name, member info, and primary color
❌ 修正した / htmlいじった
🚀

git push

ローカルの変更をリモートに送信する
git push は、ローカルでコミットした変更を GitHub などのリモートリポジトリにアップロードするコマンドです。 チームメンバーがあなたの変更を見られるようになります。
📋 CONCEPT
コミットはローカルだけに存在します。push することで初めてリモートに反映され、他の人と共有できます。push したブランチは GitHub 上で Pull Request を作成できます。
  • ✓
    ヒストリーでコミットを確認する
    Sourcetree 上部の ヒストリー タブを開き、先ほどのコミットが表示されていることを確認します。
  • ✓
    「プッシュ」ボタンをクリックする
    上部ツールバーの プッシュ ボタンをクリックします。
  • ✓
    プッシュするブランチを確認する
    ダイアログで feature/yourname ブランチが選択されていることを確認して「プッシュ」をクリックします。
  • ✓
    GitHub でプッシュを確認する
    ブラウザでリポジトリを開き、feature/yourname ブランチが表示されることを確認します。
ここで GitHub を開いて、ブランチが追加されているか確認しましょう。「Compare & pull request」ボタンも確認できます。
⚠️ CAUTION
git push -f（強制プッシュ）は他の人のコミットを消してしまう危険なコマンドです。チームでの作業では絶対に使わないようにしましょう。
⬇️

git pull

リモートの最新変更をローカルに取り込む
git pull は、リモートリポジトリの最新の変更をローカルに取り込むコマンドです。 チームメンバーがプッシュした変更を自分のPCに反映させるために使います。毎朝作業を始める前に実行するのが習慣です。
📋 CONCEPT
git pull = git fetch（リモートの変更を取得）+ git merge（ローカルに統合）の2つの操作をまとめたコマンドです。
  • ✓
    main ブランチに切り替える
    左サイドバーの main をダブルクリックして切り替えます。
  • ✓
    「プル」ボタンをクリックする
    上部ツールバーの プル ボタンをクリックします。
  • ✓
    プルするブランチを確認して実行する
    ダイアログで origin/main が選択されていることを確認し「プル」をクリックします。
  • ✓
    ヒストリーを確認する
    他のメンバーのコミットが ヒストリー に表示されることを確認します。
  • ✓
    【ボーナス】コンフリクトを体験する（任意）
    同じファイルの同じ行を2人で変更すると「コンフリクト」が発生します。講師の指示に従って解決方法を確認しましょう。
💡 BEST PRACTICE
作業を始める前に必ず git pull を実行しましょう。リモートと差異がある状態で作業を続けると、コンフリクトが起きやすくなります。
