# Git 基礎演習 - 解答と解説

## 演習の完全な実行手順

> **注意**: `Challenge`フォルダはすでに Git リポジトリとして初期化されています。`cd Challenge`で移動して、すぐに演習を始められます。

### ステップ 1: 最初のコミット

```bash
cd Challenge
git status
git add index.html
git add style.css
git status
git commit -m "初回コミット: HTMLとCSSファイルを追加"
git log
```

**解説:**

- `git status`: 現在のリポジトリの状態を表示します。最初は追跡されていないファイル（Untracked files）として`index.html`と`style.css`が表示されます。
- `git add <ファイル名>`: 指定したファイルをステージングエリアに追加します。ステージングエリアは、次のコミットに含める変更を準備する場所です。
- `git add .`: カレントディレクトリ以下のすべての変更をステージングします（便利なショートカット）。
- `git commit -m "メッセージ"`: ステージングされた変更をコミット（記録）します。`-m`オプションでコミットメッセージを指定します。
- `git log`: コミット履歴を表示します。コミット ID、作成者、日時、メッセージが表示されます。

**重要なポイント:**

- コミットメッセージは変更内容が明確にわかるように書きましょう
- 1 つのコミットには関連する変更をまとめます

---

### ステップ 2: HTML の修正

```bash
# エディタでindex.htmlを開き、以下のように変更:
# <h1>Git演習ページ</h1> → <h1>Gitワークショップへようこそ！</h1>

git status
git diff
git add index.html
git commit -m "タイトルを更新"
```

**解説:**

- `git diff`: ワーキングディレクトリとステージングエリアの差分を表示します。どの行が変更されたかを確認できます。
  - 赤色（`-`）: 削除された行
  - 緑色（`+`）: 追加された行
- ファイルを変更した後、`git status`で「Changes not staged for commit」と表示されます。
- `git add`でステージングすると、「Changes to be committed」に移動します。

**ベストプラクティス:**

- コミット前に`git diff`で変更内容を確認する習慣をつけましょう
- 意図しない変更が含まれていないかチェックします

---

### ステップ 3: CSS の修正

```bash
# エディタでstyle.cssを開き、以下のように変更:
# background-color: #f5f5f5; → background-color: #e3f2fd;

git status
git diff style.css
git add style.css
git commit -m "背景色を青系に変更"
```

**解説:**

- `git diff <ファイル名>`: 特定のファイルの差分のみを表示します。
- 複数のファイルを変更している場合、個別に確認できて便利です。

---

### ステップ 4: 複数ファイルの同時変更

```bash
# index.htmlに以下の行を追加:
# <p>Gitはバージョン管理システムで、コードの変更履歴を記録し、チームでの開発を効率化します。</p>

# style.cssで以下を変更:
# h1のfont-sizeを追加: font-size: 2rem;
# pのfont-sizeを追加: font-size: 1.1rem;
# pのmargin-bottomを追加: margin-bottom: 15px;

git status
git add .
git commit -m "コンテンツとスタイルを追加"
```

**解説:**

- `git add .`: カレントディレクトリ以下のすべての変更をまとめてステージングします。
- 複数のファイルを変更した場合でも、関連する変更であれば 1 つのコミットにまとめることができます。
- `git status`で複数のファイルが「Changes not staged for commit」に表示されます。

**注意点:**

- 関連しない変更は別々のコミットに分けるべきです
- 例: バグ修正と新機能追加は別のコミットにする

---

### ステップ 5: コミット履歴の確認

```bash
git log
git log --oneline
git log --graph --oneline
```

**解説:**

- `git log`: 詳細なコミット履歴を表示（コミット ID、作成者、日時、メッセージ）
- `git log --oneline`: 1 行で簡潔に表示（コミット ID の短縮版とメッセージのみ）
- `git log --graph --oneline`: ブランチの分岐をグラフで視覚化（この演習では 1 本線）

**その他の便利なオプション:**

- `git log -n 5`: 最新 5 件のみ表示
- `git log --author="名前"`: 特定の作成者のコミットのみ表示
- `git log --since="2024-01-01"`: 特定の日付以降のコミットのみ表示

### ステップ 6: ブランチの作成と確認（オプション）

```bash
git branch feature/basic-title
git branch
git checkout feature/basic-title
git checkout main
```

**解説:**

- `git branch <ブランチ名>`: 新しいブランチを作成します（現在のブランチから分岐）。
- `git branch`: ブランチ一覧を表示します。現在のブランチには`*`が表示されます。
- `git checkout <ブランチ名>`: 指定したブランチに切り替えます。
- このステップでは、ブランチの作成と切り替えの基本操作を体験します。

**ブランチとは:**

- ブランチは独立した開発ラインです
- メインブランチ（`main`または`master`）を壊すことなく、新しい機能を開発できます
- 詳細なブランチ操作は応用演習で学びます

**重要なポイント:**

- ブランチを作成しても自動的に切り替わりません
- 切り替えるには`git checkout`または`git switch`が必要です
- ブランチ名は意味のある名前にしましょう（例: `feature/basic-title`, `bugfix/header`）

### ステップ 7: 変更の取り消し

```bash
# index.htmlに適当な変更を加える（例: タイトルを変更）

git status
git restore index.html
git status
```

**解説:**

- `git restore <ファイル名>`: ワーキングディレクトリの変更を取り消し、最後のコミットの状態に戻します。
- **注意**: この操作は取り消せません！変更が完全に失われます。
- ステージング済みの変更を取り消す場合は`git restore --staged <ファイル名>`を使います。

**旧バージョンの Git:**

- Git 2.23 以前では`git checkout -- <ファイル名>`を使用していました
- `git restore`はより直感的な新しいコマンドです

## Git の基本的なワークフロー

```
ワーキングディレクトリ → ステージングエリア → リポジトリ
     (編集)          git add        git commit
```

1. **ワーキングディレクトリ**: ファイルを編集する場所
2. **ステージングエリア**: 次のコミットに含める変更を準備する場所
3. **リポジトリ**: コミットされた変更が永続的に保存される場所

## よく使う Git コマンドまとめ

| コマンド                          | 説明                       |
| --------------------------------- | -------------------------- |
| `git status`                      | 現在の状態を確認           |
| `git add <ファイル>`              | ファイルをステージング     |
| `git add .`                       | すべての変更をステージング |
| `git commit -m "メッセージ"`      | 変更をコミット             |
| `git log`                         | コミット履歴を表示         |
| `git log --oneline`               | 簡潔な履歴を表示           |
| `git diff`                        | 変更内容を表示             |
| `git diff <ファイル>`             | 特定ファイルの変更を表示   |
| `git restore <ファイル>`          | 変更を取り消す             |
| `git restore --staged <ファイル>` | ステージングを取り消す     |

## コミットメッセージのベストプラクティス

### 良い例:

- `初回コミット: HTMLとCSSファイルを追加`
- `タイトルを更新`
- `背景色を青系に変更`
- `ユーザー登録機能を実装`
- `バグ修正: ログイン時のエラーを解消`

### 悪い例:

- `修正` （何を修正したか不明）
- `aaa` （意味不明）
- `いろいろ変更` （具体性がない）
- `テスト` （何をテストしたか不明）

### ルール:

1. 何を変更したかを明確に書く
2. 簡潔に（50 文字以内が理想）
3. 命令形で書く（「〜を追加」「〜を修正」）
4. 英語の場合は動詞から始める（Add, Fix, Update, Remove など）

## トラブルシューティング

### Q: `git add`を間違えてしまった

A: `git restore --staged <ファイル名>`でステージングを取り消せます

### Q: コミットメッセージを間違えた

A: `git commit --amend -m "新しいメッセージ"`で最後のコミットメッセージを修正できます

### Q: まだコミットしていない変更を取り消したい

A: `git restore <ファイル名>`で変更を取り消せます（注意: 復元できません）

### Q: コミット履歴を見やすくしたい

A: `git log --oneline --graph --all --decorate`を使うか、GUI ツール（GitKraken、SourceTree など）を使用します

### Q: `.git`フォルダを削除してしまった

A: すべての Git 履歴が失われます。バックアップがない限り復元できません。
